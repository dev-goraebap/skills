# 토론 — 범위 확정

막연한 아이디어를 구체적인 빌드 범위로 좁히는 과정. AI는 질문자가 아니라 **대화 파트너**로서 방향을 함께 찾아간다.

---

## 토론 흐름

### 1단계: 컨텍스트 파악

아래 질문을 순서대로 묻되, 사용자가 이미 답한 내용은 생략한다.

> "어떤 제품·서비스를 만들고 계신가요? 30초 분량으로 설명해주세요."

> "이번 MVP로 **딱 하나** 보여주고 싶은 핵심 가치나 기능이 있다면 무엇인가요?"

> "누가 볼 예정인가요? (팀원, 클라이언트, 투자자, 내부 검토…)"

> "지금 가장 불확실한 부분이 어디인가요? 이 MVP로 그 불확실성을 해소하고 싶으신가요?"

---

### 2단계: 범위 제안

사용자의 답변을 바탕으로 AI가 먼저 범위를 제안한다. 사용자가 직접 범위를 정하도록 기다리지 않는다.

**제안 템플릿:**

> 말씀하신 내용을 기반으로 이렇게 범위를 잡으면 어떨까요?
>
> **만들 것**
> - [핵심 화면 1-2개]
> - [주요 인터랙션 1가지]
> - 목업 데이터로 해피 패스만
>
> **만들지 않을 것**
> - 실제 API 연결
> - 로그인·인증
> - 에러 처리
> - [사용자가 언급했지만 핵심이 아닌 것]
>
> 이 방향으로 진행할까요?

---

### 3단계: 확정

사용자가 동의하면 한 문장으로 범위를 정리하고 구현으로 넘어간다.

> "확인했습니다. **[한 문장 요약]** — 이 기준으로 바로 만들어볼게요."

수정 요청이 있으면 반영 후 다시 확인한다.

---

## AI가 주도해야 하는 이유

사용자는 종종 범위를 스스로 좁히지 못한다. "이것도 필요하고 저것도 필요한데…"라는 상태에서 출발한다. AI가 명확한 제안을 먼저 내놓아야 대화가 진행된다.

- 사용자가 주저하면: 더 작게 제안한다
- 사용자가 기능을 추가하려 하면: "이번 MVP에서 꼭 필요한가요? 다음 버전에서 검증하는 건 어떨까요?"라고 되묻는다
- 불확실성이 크면: "가장 핵심이 되는 화면 하나만 만들어봐요"로 유도한다

---

## 범위 확정 체크리스트

구현에 들어가기 전 아래를 확인한다.

- [ ] 보여줄 화면이 몇 개인지 확정됐다 (권장: 1~2개)
- [ ] 핵심 사용자 흐름이 한 줄로 정리됐다
- [ ] 목업 데이터로 대체할 부분이 명확하다
- [ ] 사용자가 "이 방향 맞아요"를 확인했다
