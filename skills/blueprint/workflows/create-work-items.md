# 작업 아이템 생성 가이드라인

- 작업의 규모는 사용자와의 합의 지점이기 때문에 각 파일마다 다르다.
- 작업은 새로운 기능을 추가하는 것 뿐만 아니라, 버그수정, 기존의 기능 변경 등 다양하다.
- **승인된 작업 아이템은 절대 수정하지 않는다.** 승인 이후의 작업 아이템은 이력(히스토리)이다. 요구사항이 변경되더라도 기존 작업 아이템은 그대로 두고, 새로운 작업 아이템을 생성한다.

## 파일명 규칙

형식: `v<major>.<minor>.<patch>-<type>-<설명>.md`

### 버전 규칙

| 구분 | 의미 | 예시 |
|------|------|------|
| **major** | 첫 릴리스/출시. v0.x.x는 모두 개발 단계 | `v1.0.0` = 첫 프로덕션 릴리스 |
| **minor** | 새로운 기능 추가 또는 구조적 변경 | `v0.2.0` = 두 번째 기능 단위 작업 |
| **patch** | 기존 기능의 버그 수정, 소규모 개선 | `v0.2.1` = v0.2.0에 대한 수정 |

### type 키워드

파일명의 type은 frontmatter의 `issue-type`과 일치해야 한다.

| type | 설명 | 파일명 예시 |
|------|------|-------------|
| `mvp` | 초기 핵심 기능 전체 구현. 범위가 넓음 | `v0.1.0-mvp.md` |
| `feature` | 단일 기능 추가 | `v0.2.0-feature-notification.md` |
| `fix` | 버그 수정 | `v0.2.1-fix-notification.md` |
| `refactor` | 기능 변경 없이 코드 구조 개선 | `v0.3.0-refactor-auth-module.md` |
| `setup` | 프로젝트 초기 설정, 환경 구성 | `v0.1.0-setup-project.md` |
| `separation` | 서비스/모듈 분리 | `v0.7.0-separation-backend.md` |

위 키워드 외에도 작업 성격에 맞는 키워드를 사용할 수 있다 (예: `migration`, `docs`, `perf` 등).

## 양식

```markdown
---
type: work-items
issue-type: <파일명의 type과 동일>
status: draft
created-at: yyyy-mm-dd
---

## 관련 파일

- `(.blueprint/<scope>/design/lofi-design-tokens.md)` - 이 작업에서 참고할 설계 문서 (해당되는 경우에만).
- `path/to/file1.ts` - 이 파일이 관련된 이유에 대한 간략한 설명.
- `path/to/file1.test.ts` - `file1.ts`의 유닛 테스트.

## 작업 완료 지침

**중요:** 각 작업을 완료할 때마다 `- [ ]`를 `- [x]`로 변경하여 체크한다. 상위 작업 전체가 아닌 각 하위 작업을 완료할 때마다 업데이트한다.

## 작업 목록

- [ ] 0.0 기능 브랜치 생성
  - [ ] 0.1 새 브랜치를 생성하고 체크아웃한다 (예: `git checkout -b feature/[기능명]`)
- [ ] 1.0 상위 작업 제목 (FR-001)
  - [ ] 1.1 [하위 작업 설명]
  - [ ] 1.2 [하위 작업 설명]
- [ ] 2.0 상위 작업 제목 (FR-002)
  - [ ] 2.1 [하위 작업 설명]

```
