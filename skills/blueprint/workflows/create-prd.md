# 규칙: PRD(제품 요구사항 문서) 생성

## 목표

사용자의 초기 프롬프트를 기반으로 마크다운 형식의 상세한 PRD를 작성하도록 AI 어시스턴트를 안내한다. PRD는 명확하고 실행 가능하며, 주니어 개발자도 이해하고 구현할 수 있을 정도로 작성되어야 한다.

## 프로세스

1.  **초기 프롬프트 수신:** 사용자가 새로운 기능이나 요구사항에 대한 간략한 설명을 제공한다.
2.  **명확화 질문:** PRD 작성 전, *반드시* 명확한 PRD 작성에 필요한 핵심 질문만 한다. 질문은 3~5개의 중요한 이해 공백에 한정한다. 기능의 "무엇"과 "왜"를 이해하는 것이 목표이며, "어떻게"는 개발자가 결정할 부분이다. 사용자가 쉽게 답변할 수 있도록 선택지를 번호/문자 목록으로 제공한다.
3.  **PRD 생성:** 초기 프롬프트와 사용자의 답변을 기반으로 아래 구조에 따라 PRD를 생성한다.
4.  **PRD 저장:** 생성된 문서를 `prd-[기능명].md`로 저장한다.

## 명확화 질문 (가이드라인)

명확한 PRD 작성에 필요한 핵심 질문만 한다. 초기 프롬프트가 모호하거나 필수 맥락이 누락된 부분에 집중한다. 일반적으로 명확화가 필요한 영역:

*   **문제/목표:** 불명확할 경우 - "이 기능이 사용자에게 어떤 문제를 해결해주나요?"
*   **핵심 기능:** 모호할 경우 - "사용자가 수행할 수 있어야 하는 핵심 동작은 무엇인가요?"
*   **범위/경계:** 광범위할 경우 - "이 기능이 하지 *않아야* 할 것이 있나요?"
*   **성공 기준:** 명시되지 않은 경우 - "이 기능이 성공적으로 구현되었는지 어떻게 알 수 있나요?"

**중요:** 초기 프롬프트에서 합리적으로 추론 가능한 경우에는 질문하지 않는다. PRD의 명확성에 큰 영향을 미치는 질문을 우선시한다.

### 형식 요구사항

- **모든 질문에 번호를 매긴다** (1, 2, 3 등)
- **각 질문의 선택지를 A, B, C, D 등으로 나열한다**
- 사용자가 "1A, 2C, 3B"처럼 간편하게 답변할 수 있도록 한다

### 예시 형식

```
1. 이 기능의 주요 목표는 무엇인가요?
   A. 사용자 온보딩 경험 개선
   B. 사용자 유지율 향상
   C. 고객 지원 부담 감소
   D. 추가 수익 창출

2. 이 기능의 대상 사용자는 누구인가요?
   A. 신규 사용자만
   B. 기존 사용자만
   C. 모든 사용자
   D. 관리자만

3. 이 기능의 예상 일정은 어떻게 되나요?
   A. 긴급 (1~2주)
   B. 높은 우선순위 (3~4주)
   C. 일반 (1~2개월)
   D. 향후 검토 (3개월 이상)
```

## PRD 양식

```markdown
---
type: prd
status: draft
created-at: yyyy-mm-dd
updated-at: yyyy-mm-dd
---

## 소개/개요

기능과 해결하려는 문제를 간략하게 설명한다. 목표를 명시한다.

## 목표

이 기능의 구체적이고 측정 가능한 목표를 나열한다.

## 사용자 스토리

기능 사용법과 이점을 설명하는 사용자 시나리오를 상세히 기술한다.

## 기능 요구사항

각 요구사항에 `FR-001` 형식의 ID를 채번한다. 명확하고 간결한 언어를 사용한다.

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| FR-001 | 시스템은 사용자가 프로필 사진을 업로드할 수 있어야 한다. | 높음 |
| FR-002 | ... | ... |

## 비목표 (범위 외)

범위 관리를 위해 이 기능에 포함되지 *않을* 사항을 명확히 기술한다.

## 설계 고려사항 (선택)

목업 링크, UI/UX 요구사항, 관련 컴포넌트/스타일을 기술한다.

## 기술 고려사항 (선택)

알려진 기술적 제약, 의존성, 제안을 기술한다 (예: "기존 Auth 모듈과 통합해야 함").

## 성공 지표

이 기능의 성공을 어떻게 측정할 것인지 기술한다 (예: "사용자 참여도 10% 증가", "X 관련 지원 티켓 감소").

## 미결 사항

추가 명확화가 필요한 남은 질문이나 영역을 나열한다.
```

## 대상 독자

PRD의 주요 독자는 **주니어 개발자**로 가정한다. 따라서 요구사항은 명시적이고 모호하지 않아야 하며, 가능한 한 전문 용어를 피한다. 기능의 목적과 핵심 로직을 이해할 수 있을 정도로 충분한 세부 사항을 제공한다.
