# PRD 변경 가이드라인

## 목적

개발 착수 이후 제품 요구사항이 변경될 때 PRD 문서를 갱신하는 규칙을 정의한다.

초안 검토 단계의 수정은 단순한 드래프팅이므로 이 규칙의 대상이 아니다. 이 규칙은 PRD가 `approved` 상태인 이후에만 적용된다.

## 트리거 시점

워크플로우 2(소프트웨어 개발) 진행 중 다음 상황이 감지되면 PRD 변경이 필요하다:

- 사용자가 기존 FR-ID에 없는 작업을 요청한 경우
- 사용자의 요청이 기존 FR의 내용을 변경해야 하는 경우

## 변경 규칙

1. 기존 내용을 **삭제하지 않는다**.
2. 변경된 항목은 `~~취소선~~`으로 처리한다.
3. 새로운 내용을 취소선 바로 아래에 추가한다.
4. `updated-at`을 갱신한다.

### 변경 유형별 예시

#### 대체: 기존 기능을 새 기능으로 교체

기존 행을 취소선 처리하고, 새 ID를 채번하여 추가한다.

```markdown
| ~~FR-001~~ | ~~시스템은 이메일로만 로그인할 수 있어야 한다.~~ | ~~높음~~ |
| FR-005 | 시스템은 이메일 또는 소셜 로그인을 지원해야 한다. | 높음 |
```

#### 삭제: 기존 기능을 제거

기존 행을 취소선 처리만 한다. 새 ID는 채번하지 않는다.

```markdown
| ~~FR-002~~ | ~~시스템은 게스트 모드를 지원해야 한다.~~ | ~~낮음~~ |
```

#### 신규: 완전히 새로운 기능 추가

기존 마지막 FR 번호 다음으로 새 ID를 채번한다.

```markdown
| FR-006 | 소셜 로그인 시 프로필 정보를 자동으로 가져온다. | 중간 |
```

## 기존 작업 아이템과의 관계

- 이미 완료된 작업 아이템은 건드리지 않는다. 승인된 작업 아이템은 이력(히스토리)이다.
- PRD의 취소선으로 변경 이력이 남아 있으므로, 기존 FR-ID를 참조하는 작업 아이템도 문맥이 유지된다.
- 변경된 요구사항에 대한 구현은 새로운 작업 아이템으로 생성한다.
